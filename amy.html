The project

Building a compiler from scratch, one stage per week

Amy is a small functional programming language with algebraic data types, pattern matching, and a module system. Over 5 weeks of labs, we built a complete compiler for it in Scala, from the very first character of source code all the way to executable WebAssembly bytecode. Then, as a final extension project, we added a second backend that compiles Amy to C instead.

This was not a "plug-in a library" kind of project. Every stage was built from the ground up, each one feeding into the next.

Full compiler pipeline — 5 labs + C backend extension

The labs

One week, one stage

Wk 1
Lexer — source text → token stream
Read raw characters and group them into meaningful tokens: keywords, identifiers, literals, operators. Handle edge cases like multi-line comments, escaped strings, and invalid characters. This is the lowest level of the pipeline — no structure yet, just words.
Wk 2
Parser — token stream → raw AST
Apply grammar rules to build a tree from the token stream. Handle operator precedence, nested expressions, if/else, pattern matching syntax, and module declarations. Produce a meaningful error if the input doesn't conform to Amy's grammar.
Wk 3
Name analyser — resolve identifiers, check scopes
Walk the raw AST and resolve every name to its definition. Catch uses of undeclared variables, duplicate definitions, and module import errors. After this stage, every identifier in the tree points unambiguously to the right declaration.
Wk 4
Type checker — verify types, produce typed AST
Annotate every expression with its type and verify consistency. Handle type inference for let-bindings, check exhaustiveness of pattern matches, and reject any ill-typed program with a clear error. The typed AST is what both backends consume.
Wk 5
WebAssembly backend — typed AST → .wat bytecode
Translate the typed AST into WebAssembly text format. Map Amy's ADTs onto Wasm's linear memory, handle the stack machine's calling convention, and emit valid .wat that can be run in any Wasm runtime.
Ext.
C backend — our extension
Add a second backend that compiles the same typed AST to portable C. All the hard problems Wasm hides, ADT encoding, expression-to-statement lowering, name mangling, had to be solved explicitly. This is the bulk of the extension work, described in detail below.

The extension

Why C is harder than WebAssembly

WebAssembly is a clean compilation target : it has sum types, a structured stack, and no C-style expression/statement split. C has none of that. Adding a C backend forced us to solve every problem Wasm had conveniently abstracted away.

ADT encoding — from Amy constructors to flat C heap arrays

Hard problems

Three things C doesn't give you for free

Ⅰ
Expressions vs. statements
Amy's if/else and match are expressions, they produce values. C's if is a statement, it doesn't. We used GCC statement-expressions ({ ... }) to bridge the gap, wrapping C statements into expression context wherever needed.
Ⅱ
Safe nested pattern matching
A nested pattern like Cons(Cons(x, _), _) needs to read inner fields before the outer tag is confirmed. We guard every field load with a tag check first — if the tag fails, a safe default propagates instead of an invalid memory access.
Ⅲ
Evaluation order
Amy guarantees left-to-right argument evaluation. C doesn't. For constructor calls, we evaluate every argument into a temporary variable first, then allocate and fill the struct. This ensures consistent behaviour across all C compilers.
Ⅳ
Name mangling
Amy modules allow List.Cons and Tree.Cons to coexist. C has a flat global namespace. Every Amy identifier becomes Module_name in the generated C, a simple but essential transformation to avoid clashes.

Pattern matching lowering — matched flag + shared result variable

In practice

Amy → C in one command

Add --c to the compiler invocation — everything else stays the same.

# Compile Amy source to C
sbt "run --c ./library/Std.amy ./examples/Hanoi.amy"

# Compile generated C and run
gcc -o hanoi cout/Hanoi.c cout/amy_runtime.c
./hanoi
# → Hanoi for 4 plates: 15

The amy_runtime.c is a small handwritten C file that implements the standard library (print, read, string operations). Having two backends also gave us differential testing for free — any Amy program from the labs could be run through both backends and outputs compared.

Takeaways

What building a compiler teaches you

01
Languages are stacked abstractions. Every feature you take for granted in a language, pattern matching, type inference, closures, is someone else's lowering problem solved cleverly.
02
C's expression/statement split is a real constraint. Modern languages paper over it with tricks, statement-expressions, continuations, SSA form. Writing a backend forces you to pick one.
03
Type erasure is a superpower. Once the type checker has verified the program, generated code doesn't need types at all. Every value as intptr_t made the whole backend much simpler.
04
Two backends = free test suite. Running every lab program through both Wasm and C backends and comparing outputs caught several subtle bugs in both, a trick borrowed from differential fuzzing.
