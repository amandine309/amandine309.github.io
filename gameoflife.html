The project

Implementing Conway's Game of Life - in the language beneath all languages

The goal was to implement a fully playable version of Conway's Game of Life in RISC-V assembly, running on a real FPGA board with a 10×12 LED grid, physical buttons for control, and a 7-segment display for the step counter.

At this point in my studies, I had only ever coded in Java. Assembly was a completely different world, and the mental shift required was profound.

It felt like doing Latin. Latin is the foundation of French, and Assembly is the foundation of every high-level language I had ever used. You're not calling methods anymore. You're moving bytes.

A for loop in Java is one line. In assembly, it's a counter register, an increment, a branch condition, and a label to jump back to. Every abstraction you take for granted, variables, functions, if/else, disappears. What's left is the machine itself.

The mental shift

The same loop : Java vs. Assembly

Something as simple as counting neighbours requires completely different thinking at the assembly level.

♘ Java : high level
int count = 0; for (int dx = -1; dx <= 1; dx++) { for (int dy = -1; dy <= 1; dy++) { if (dx==0 && dy==0) continue; count += getCell( (x+dx+12)%12, (y+dy+10)%10 ); } } // 8 lines, reads like English
♗ RISC-V Assembly
# count neighbours of (a0,a1) li t0, 0 # count = 0 li t1, -1 # dx = -1 dx_loop: li t2, -1 # dy = -1 dy_loop: # skip (0,0) beq t1, x0, skip_center # compute (x+dx) mod 12 add a0, s0, t1 li t4, 12 rem a0, a0, t4 # ... 20+ more lines addi t2, t2, 1 blt t2, t3, dy_loop addi t1, t1, 1 blt t1, t3, dx_loop

Every single Java concept, loops, conditionals, modulo, function calls, becomes explicit register manipulation and jump instructions. Nothing is implicit. Nothing is automatic. You manage everything: the stack, the return address, the caller-saved registers. It is humbling, and it is eye-opening.

The game

Conway's Game of Life - four rules, infinite complexity

The Game of Life is a cellular automaton where cells on a 2D grid live or die based solely on their neighbours. It has no players, only an initial state and four deterministic rules. Despite their simplicity, they produce astonishingly complex emergent behaviour.

Underpopulation
A live cell with fewer than 2 live neighbours dies, it's isolated.
neighbours < 2 → dead
Overpopulation
A live cell with more than 3 live neighbours dies, too crowded.
neighbours > 3 → dead
Reproduction
A dead cell with exactly 3 live neighbours comes to life.
dead && neighbours == 3 → alive
Stasis
A live cell with 2 or 3 neighbours survives to the next generation.
alive && 2 ≤ neighbours ≤ 3 → alive

Game loop - GSA double-buffering → rules → mask → draw

Challenges

What made this project hard

Ⅰ
Thinking without abstractions
In Java, a variable is a name. In assembly, a variable is a register, and there are only 32 of them, some with special roles you can't touch. Running out of registers mid-function means spilling to the stack manually. Every simple operation requires a mental model of what's where at every instruction. Coming from Java, this was the single hardest adjustment.
Ⅱ
Function calls and the stack
In Java, you call a method and it just works. In RISC-V assembly, you have to manually save the return address (ra), save any registers you need to preserve across the call, call the function, restore everything, then return. The calling convention, which registers are caller-saved vs callee-saved, has to be respected exactly, or the program crashes in a completely unpredictable way. This took many debugging sessions to internalize.
Ⅲ
Torus topology with modulo
The game grid wraps around, the cell at column 11 is adjacent to the cell at column 0. In Java, this is one modulo operation. In assembly, handling negative results of modulo (-1 mod 12 should be 11, not -1) requires additional correction instructions. Every neighbour lookup involves computing (x±1+12)%12 and (y±1+10)%10 correctly, for all 8 neighbours of every cell on a 120-cell grid.
Ⅳ
MMIO - writing to hardware through memory
To light up an LED, you don't call a function, you construct a 32-bit word with the right bit pattern for column, row, color channel, and value, then store it at the right memory address. Getting one LED wrong meant debugging the bit packing manually. The LED control register alone has 5 distinct fields across 32 bits, and the draw_gsa procedure has to get every single one right for each of the 120 cells.

What was built

20 assembly procedures from scratch

The full game required implementing 20 independent assembly procedures, each tested by an automated grader. From low-level LED control to the full game loop with button handling and interrupt-driven input.

clear_leds
Reset all 120 LEDs to off
set_pixel
Light one LED at (x,y) in red
wait
Variable delay based on game speed
get_gsa
Read a row from active GSA
set_gsa
Write a row to active GSA
draw_gsa
Render full GSA to LED array
random_gsa
Fill GSA with random cells
find_neighbours
Count live neighbours of (x,y)
cell_fate

update_gsa
Compute next full game state
mask

change_speed
Inc/dec game speed 1–10

Takeaways

What assembly gave me that Java never could

01
High-level languages are not magic, they are structured assembly. Every Python list, every Java method call, every JavaScript closure ultimately becomes sequences of load, store, branch, and jump instructions. Knowing this changes how you read code.
02
The stack is not invisible. In Java the JVM manages it silently. In assembly you push and pop manually, you decide what gets saved, and a single mistake corrupts the return address and crashes the program in a completely opaque way. After this project, stack frames are never abstract.
03
Bit manipulation is a real skill. Packing 5 fields into a 32-bit LED register, extracting a single cell from a GSA word with shifts and masks, things that seem trivial in high-level code require explicit bit arithmetic in assembly. It makes you think in binary.
04
Emergence is real. Four deterministic rules applied to 120 cells produce gliders, oscillators, still lifes, and chaos. Watching the patterns evolve on the LED grid, patterns that emerge from pure logic you wrote in assembly, is genuinely beautiful.
