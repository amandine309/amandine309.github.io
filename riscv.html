The project

Building a CPU from first principles, gate by gate

Over five incremental labs, I designed and implemented a complete RISC-V multicycle processor in Verilog, from the very first arithmetic unit all the way to a fully functional CPU with memory-mapped peripherals and hardware interrupt handling.

This wasn't a simulation exercise. Each lab built on the previous one, and the final result was a real processor running on FPGA, capable of executing RISC-V assembly programs, reacting to button presses via interrupts, and controlling LEDs and 7-segment displays through memory-mapped I/O.

Understanding how a CPU actually works, not just using one, is one of the most foundational experiences in computer science. This project delivered exactly that.

Full processor architecture : 5 labs, built incrementally

Five labs

From arithmetic unit to full interrupt system

L1
ALU : the arithmetic core
Designed the ALU with four sub-units: Add/Sub (using two's complement, A−B becomes A+(~B)+1), a comparator for signed and unsigned comparisons (LT vs LTU : same bits, different meaning), a logical unit for AND/OR/XOR, and a shift unit for SLL/SRL/SRA. A final multiplexer selects the right result based on the 6-bit op code. Without this, the processor can't do anything.
L2
Multicycle processor : FSM + datapath
Built the full processor datapath around a finite state machine controller. The key insight of multicycle design: instead of doing FETCH + DECODE + EXECUTE + MEMORY + WRITEBACK in one (very slow) cycle, each stage gets its own cycle. The clock can run faster, and complex instructions simply use more cycles. Implemented all RV32I instruction types: R-type, I-type (with sign extension), LOAD/STORE, branches, JAL, JALR.
L3
External memory & MMIO peripherals
Connected the CPU to external synchronous RAM (data arrives one cycle after the address, the multicycle design handles this naturally) and implemented Memory-Mapped I/O. Peripherals(LEDs, 7-segment display, buttons) are mapped to specific address ranges. The same lw/sw instructions that access RAM are used to control hardware. A decoder examines the address and activates the right peripheral.
L4
CSR controller: special registers
Implemented the Control and Status Registers: mtvec (interrupt handler address), mepc (return address saved before jumping to handler), mcause (what triggered the interrupt), and ipending (pending interrupt flags). Added the CSR instruction set (csrrw, csrrs, csrrc and immediate variants). These registers are what allows the CPU to save its state and jump to an interrupt handler safely.
L5
Interrupts: full integration
Connected everything: button presses generate an IRQ signal, the controller detects rising/falling edges (by comparing current and previous state), the CPU saves PC to mepc, jumps to mtvec, executes the handler, reads the SRC register to identify the source, clears it, and returns via mret. The processor can now multitask, running a main loop and reacting to hardware events without polling.

How the FSM works

The multicycle state machine

The controller is a finite state machine. At each clock cycle it moves to a new state and activates the right control signals. A LOAD instruction takes more cycles than an ADD because it needs to access memory, and that's exactly the point of multicycle design.

Cycle 1
FETCH1
Put PC on the address bus. RAM is synchronous, the data will arrive next cycle.
Cycle 2
FETCH2
Store the instruction in IR. Increment PC ← PC + 4.
Cycle 3
DECODE
Read opcode, funct3, funct7. Determine instruction type and next state.
Cycle 4
EXECUTE
ALU computes result. For branches: evaluate condition. For R/I-type: compute value.
Cycle 5 (LOAD)
LOAD1
Compute memory address (rs1 + imm), send to RAM.
Cycle 6 (LOAD)
LOAD2
Data arrives from RAM. Write to destination register rd.

Multicycle FSM : LOAD takes 6 cycles, ADD takes 4

Memory-Mapped I/O

Peripherals as memory addresses

The elegant insight of MMIO: instead of special I/O instructions, every peripheral gets an address in the memory space. The CPU uses the exact same lw / sw instructions to control LEDs or read button states as it does to access RAM. A decoder module examines the address and activates the right peripheral, all others stay inactive.

Address Peripheral Operation
0x50000000 LEDs sw writes bit pattern → controls which LEDs light up
0x60000000 7-segment display sw writes encoded digit → displayed on screen
0x70000000 Buttons / Switches lw reads current button state (10 buttons + 8 switches)
0x80000000 RAM Standard synchronous memory : program + data

Interrupt handling

The full interrupt sequence : 8 steps

Without interrupts, the CPU would waste cycles constantly polling buttons in a loop. With interrupts, it runs its main program and only reacts when hardware signals it. Here's what happens from button press to program resume:

1
Button pressed → generates IRQ
2
Controller detects rising edge
3
mepc ← current PC saved
4
PC ← mtvec (handler addr)
5
Handler starts executing
6
Read SRC, handle event, clear SRC
7
mret executed
8
PC ← mepc, program resumes

Edge detection is worth highlighting: to detect a button press (0→1 transition), the controller stores the previous state and compares it to the current one. If old=0 and new=1, it's a rising edge, fire the interrupt. This is how the SRC register gets set, and why clearing it at the end of the handler is critical to avoid re-triggering.

Takeaways

What building a CPU from scratch teaches you

01
Everything is abstraction on abstraction. An instruction like "add x1, x2, x3" hides a fetch cycle, a decode cycle, an ALU operation, and a register write, each a separate hardware event. Building it removes the magic.
02
Timing is everything in hardware. Unlike software where a function call just works, in Verilog you think in clock cycles. A synchronous RAM means you ask for data in cycle N and get it in cycle N+1, that's a design decision, not a bug.
03
MMIO is elegant. The same instruction set that operates on integers in registers also controls physical LEDs, just by targeting a different address. This unification is why RISC architectures are clean to implement.
04
Interrupts are the foundation of reactive systems. Every OS, every embedded device, every real-time system relies on this mechanism. Implementing it in hardware, saving state, jumping, restoring, makes it concrete forever.
