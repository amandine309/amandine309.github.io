The project

An autonomous robot that maps its world entirely on-chip

SLAMCar is a mobile robot built on a modified RC car that autonomously maps its environment, avoids obstacles, and navigates to user-defined goals, all in real time, with no external compute. No PC for processing, no cloud offloading. Everything from SLAM, path planning, to obstacle avoidance runs on a dual-core ESP32 microcontroller.

This was the central constraint that drove every technical decision we made: make it fully embedded or don't bother.

How the mapping works

TinySLAM on an embedded chip : the full pipeline

The core challenge was running Simultaneous Localization and Mapping on a microcontroller with ~520KB of RAM and no FPU. Here's how the pipeline works:

01
LiDAR scan acquisition & downsampling
Each scan is capped to 180 beams, keeping only the 120 most useful. During particle weighting, we further subsample every 3rd beam to cut compute cost while preserving good angular coverage. All ranges outside [0.05m – 6.0m] are discarded as noise.
02
Particle filter, the prediction step
Each of the 100 particles is propagated using a constant-velocity motion model with Gaussian noise on translation and rotation. Since we had no wheel encoder (the motor has no steps), the motion model relied entirely on LiDAR matching which made drift a real problem.
03
Particle weighting — scan-to-map matching
Each LiDAR beam is projected into world coordinates using the particle's pose, then into grid coordinates. We read the occupancy value and convert it to a cost: unknown ≈ 0.5 (neutral), free → high cost (bad match), occupied → low cost (good match). The average cost becomes the particle score, converted to weight via w = exp(−score / σ).
04
Resampling & pose estimation
When the effective particle count Neff drops below 50%, systematic resampling is triggered. The current robot pose is taken as the highest-weight particle. This is how we correct localization drift in absence of odometry.
05
Occupancy grid update
For each valid beam, a Bresenham line is traced from the robot cell to the hit cell. Intermediate cells are updated toward free (decremented), the hit cell toward occupied (incremented). Values are clamped on a small discrete scale to keep memory tight and avoid float arithmetic.

Real difficulties

The hard parts nobody talks about

This project looked feasible on paper. In practice, every constraint compounded the others.

Ⅰ
2 cores, no room to breathe
Core 1 handled safety, motor control and communication. Core 0 ran SLAM and path planning. There was no slack, any spike in the SLAM loop delayed navigation. The memory budget analysis I ran upfront was critical to even know if it was feasible.
Ⅱ
LiDAR-only localization
Our motor had no encoder, no odometry data at all. The robot had to localize purely from LiDAR scan matching. One ambiguous corridor and the particle filter would diverge. This forced aggressive downsampling that hurt map precision.
Ⅲ
Map size vs. resolution tradeoff
A 20m×20m map at 20cm resolution fills a significant chunk of the ESP32's RAM. We wanted finer resolution for accurate navigation but couldn't afford it. The final map was functional but visibly imprecise in complex environments.
Ⅳ
Everything must fit on-chip
No cloud, no PC assist. The SLAM algorithm, particle filter, obstacle avoidance, TCP communication and navigation algorithm all had to coexist in 520KB of RAM. I spent considerable time estimating memory footprints before writing a single line of algorithm code.

Web interface

A live dashboard for the robot

I also built the web application that lets you monitor and control the car in real time. The map updates every 2 seconds directly from the ESP32 over TCP/Wi-Fi. From the dashboard you can watch the occupancy grid build up live, switch to manual keyboard control, set a navigation destination by clicking on the map, or trigger an emergency stop.

Takeaways

What I took away from this project

01
Embedded constraints force you to really understand your algorithm. You can't hide behind compute, you have to know exactly how much memory and CPU cycles each step costs before you write it.
02
Sensor diversity matters. Relying on a single sensor (LiDAR only) for both mapping and localization created a single point of failure. A wheel encoder would have made localization far more robust.
03
Probabilistic methods work on real hardware. Seeing a particle filter converge on an actual robot navigating a real corridor was one of the most satisfying moments of the project.
04
Multi-core synchronization is subtle. Sharing state between Core 0 and Core 1 via the gstate structure required careful design to avoid race conditions without expensive locking.
