The project

A fault-tolerant distributed storage system, built from scratch in C

DKVS is a simplified implementation of Amazon Dynamo: a system that stores key-value associations redundantly across multiple servers and keeps working even when some of them fail. The interface is minimal — put(key, value) and get(key) — but the mechanics behind it involve consistent hashing, quorum consensus, and concurrent UDP networking.

Everything is written in C, using raw POSIX sockets and pthreads. No framework, no abstraction library. The whole system was built layer by layer over 8 weeks, each week adding one concept on top of a working system.

01 — Consistent hashing

The SHA-1 ring

Both server nodes and keys are projected onto a 160-bit ring by computing their SHA-1. A node's hash is derived from the string "IP Port NodeID". To find the servers responsible for a given key, the client computes the key's SHA-1 and walks clockwise around the ring until it has found N distinct physical servers, skipping nodes that map to a server already contacted.

This means one physical server can appear at multiple positions on the ring, spreading its responsibility across different key ranges and improving load balance.

Key facts
  • Node SHA-1 is computed from "127.0.0.1 1234 2" for node 2 of server at port 1234
  • Multiple nodes per server configured in servers.txt
  • Ring is sorted by SHA-1 at startup via node_list_sort()
  • Lookup wraps around the ring end back to the beginning

02 — Consensus

Quorum : trading consistency for availability

Three parameters control every read and write. Their values determine how many server failures the system can absorb while still making progress.

N
Replication factor
Maximum servers contacted per operation. Determines how many copies of each value exist in the ring.
W
Write quorum
Minimum positive acks for a successful put. With N=3 and W=2, one server failure is silently tolerated on write.
R
Read quorum
Minimum identical replies for a successful get. Conflicting values are resolved by majority vote on the client side.

What happens when servers disagree? After a partial write (W < N), some servers may hold a stale value. On the next read, the client collects up to N replies and counts occurrences of each distinct value in a local hash table. The first value to reach R identical replies wins.

If N servers are exhausted without any value reaching quorum — because too many failed or because stale values split the vote — the operation fails and returns an error.

03 — Architecture

Three layers, one UDP socket at a time

The system separates three concerns that are layered cleanly on top of each other. Building them incrementally meant each layer could be tested in isolation before the next one depended on it, the fake network module was the most important design decision of the whole project.

Layers
  • Ring layer — SHA-1 projection, sorted node list, clockwise lookup with distinct-server filtering
  • Network layer — raw UDP, per-operation socket, timeout-based receive, binary framing
  • Local layer — in-memory hash table, detached pthreads, mutex-guarded access

04 — Build process

Week by week, one abstraction at a time

Each week introduced exactly one new concept on top of a working, tested system. No week broke the previous one, that constraint shaped every design decision.

Week 6
Local server — S=N=R=W=1
Hash table, CLI tools (get, put), argument parsing, fake network layer for testing without real sockets.
Week 8
Value operations + consistent hash ring
find, substr, cat commands. SHA-1 ring: sorted node list, clockwise lookup, multi-node servers parsed from servers.txt.
Week 9
Real UDP networking
POSIX UDP sockets replace the fake layer. Binary message framing, timeout-based receive. Server-side get/put handlers.
Week 11
Quorum & voting
Full N/W/R support. Client counts replies per distinct value in a local hash table. Stops as soon as R identical replies arrive, fails if N servers are exhausted first.
Week 12
Ring monitoring — dkvs-dump-ring
New tool that walks all ring nodes in SHA-order and dumps each server's contents. Htable_dump() sends content in multiple UDP chunks. Server deduplication via local hash table.
Week 13
Parallelism
Client sends all N messages before collecting replies. Per-operation sockets prevent cross-request contamination. Server spawns a detached pthread per request; mutex guards all hash table accesses.

05 — Real difficulties

The parts that required the most care

Distributed systems have a way of making simple things complicated. A few moments that forced careful thinking.

Ⅰ
Stale replies across operations
A slow server replying to an old request would inject a stale datagram into a new operation's socket. Fix: open a fresh socket per operation, not per server — the OS assigns a new port each time, and old replies arrive on the now-closed socket.
Ⅱ
Chunked dumps over UDP
A server's hash table may not fit in a single UDP datagram. Htable_dump() takes a start-bucket index and writes as many entries as fit, returning the next unwritten index. The server loops and sends one message per chunk.
Ⅲ
Consistent hashing edge cases
A physical server appears at multiple ring nodes — the lookup must skip nodes from already-visited servers (same IP and port), not just duplicate node hashes. The ring also wraps: when the key's SHA-1 is near the end, lookup continues from the beginning.
Ⅳ
Mutex scope on the server
Each server thread reads and writes the shared hash table. The mutex must cover only the table access — not the entire request handler including the UDP reply — otherwise threads block each other during the network round-trip, defeating the point of multithreading.

06 — Takeaways

What this project taught us

01
Eventual consistency is intuitive in theory. Implementing it means reasoning about every failure mode: dropped packets, partial writes, stale reads, concurrent threads racing on shared state.
02
A fake network layer is the most important design decision you can make early. It kept every layer testable in isolation and made debugging much easier than if we'd gone straight to real sockets.
03
The same abstraction can appear at multiple levels. The hash table used locally for key-value storage also served as the vote-counting mechanism at the client level — an elegant reuse across two completely different parts of the system.
04
Multithreading forces precision. Which data is shared, which is per-thread, and for exactly how long the lock must be held — not a question you can answer approximately.
