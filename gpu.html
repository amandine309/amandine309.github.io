The project

Three assignments, one obsession: making it faster

Over three assignments, we optimized a single operation Reduced Matrix Multiplication (RMM) from a sequential C baseline all the way to a CUDA kernel running on an NVIDIA V100 GPU. Each assignment introduced a new parallelism model: shared-memory threads with OpenMP, distributed processes with MPI, then GPU compute with CUDA.

The results were striking every time. A few pragmas, a broadcast, a kernel launch, and suddenly the same computation that took 4 seconds runs in 0.1. Watching execution time collapse with each optimization was one of the most viscerally satisfying experiences I've had in a CS course.

"Two lines of code. ×6.6 speedup. That's the moment you understand why people care about parallelism."
Adding #pragma omp parallel for collapse(2) to the RMM loop — Assignment 1

The operation

What is Reduced Matrix Multiplication?

Standard matrix multiplication computes C = A × B where C[i][j] = Σ A[i][k] × B[k][j]. RMM is a variant that "reduces" the output dimensions, each output element C[i][j] aggregates contributions from a 2×2 block of A and B, combining four products into one result.

This makes every output element fully independent of every other, a property that makes RMM embarrassingly parallelizable and a perfect playground to explore different parallelism models.

RMM — 4 products per k, fully independent output elements

Three assignments

The optimization journey

Each assignment layered a new parallelism model on top of the previous one. Here's how performance evolved.

A1
OpenMP — shared-memory threads
A single pragma on the two outer loops. Each thread gets a slice of the output matrix to compute independently. We added B matrix transposition to fix cache-unfriendly column access, and split the 4 accumulators to let the compiler vectorize with SIMD. Sequential baseline: 4.4s. With OpenMP + optimizations at 8 threads: 0.41s.
×10.7 vs sequential
A2
MPI — distributed processes across nodes
Matrix B is broadcast to every process. Each process receives a contiguous chunk of rows of A via MPI_Scatterv, computes its part of C independently, then results are gathered. No communication during computation, each process is self-sufficient. At 16 processes (M=N=K=4096): ~0.6s. Best hybrid combo (2 MPI × 72 OMP threads): consistently fastest.
Linear scaling to ~16 processes
A2+
OpenMP + MPI hybrid — best of both worlds
The optimal configuration was 2 MPI processes × 72 OpenMP threads, one MPI process per physical node, OpenMP exploiting all 72 cores within each node via shared memory. This avoids inter-node communication for the inner loop while still distributing work across nodes. At M=N=K=8192: 1.97s, the fastest of all tested configurations.
Best config: 2×72 hybrid
A3
CUDA — thousands of GPU threads
One CUDA thread per output element C[i][j]. The thread computes the full k-reduction for that single element. We iteratively refined the kernel through 6 versions: better block shapes, register reuse, algebraic factorization (4 mults → 1), shared-memory tiling. V1 naive: 0.907s compute. V5 tiled: 0.122s. An 86% reduction just from memory optimizations.
−86% compute time V1→V5

Key speedup milestones across the three assignments

How GPUs work

SIMT — Single Instruction, Multiple Threads

Before this project, GPU programming was a black box. Writing CUDA forced me to understand exactly how a GPU executes code, and it's fundamentally different from a CPU. The key model is SIMT: the GPU groups 32 threads into a warp, and all 32 execute the same instruction simultaneously on different data. This is what makes GPUs so fast for matrix operations and what makes some code patterns catastrophically slow.

SIMT execution model — warps of 32 threads execute in lockstep

This has a critical consequence: if 32 threads in a warp access memory in a coalesced pattern (consecutive addresses), they all get served in one memory transaction. If they access scattered addresses, you get 32 separate transactions, a 32× slowdown. This is exactly why block shape matters so much in V2, and why transposing matrix B is such a big win: it turns scattered column accesses into sequential row reads.

CUDA optimization path

Six versions, one kernel, −88% compute time

Version What changed Compute time (8192) Δ vs V1
V0 Sequential CPU baseline — —
V1 Naive CUDA — 1 thread per output element, default block shape 0.907s baseline
V2 Better block shape (32×16 wins) — warp-aligned column access, more concurrent blocks 0.791s −12.8%
V3 Register reuse — load a0, a1, b0, b1 once per iteration instead of recomputing 0.385s −57.6%
V4 Algebraic factorization — 4 multiplications per k → 1 multiplication per k 0.322s −64.5%
V5 Shared-memory tiling — blocks load reusable tiles of A and B into fast on-chip memory 0.122s −86.6%
V6 #pragma unroll on inner k loop — reduces loop overhead but increases register pressure 0.104s −88.6%

The biggest single jump is V4→V5: shared-memory tiling. Instead of every thread fetching from slow global memory on every iteration, a whole block cooperates to load a tile of A and B into shared memory (fast, on-chip), then each thread reuses those values. This is the fundamental trick behind high-performance GEMM on GPUs, and seeing it drop compute time from 0.32s to 0.12s made it click immediately.

Shared-memory tiling — the biggest single optimization in the kernel

Takeaways

What three assignments of optimization taught me

01
Parallelism is a spectrum. OpenMP gives you threads for free. MPI distributes across machines. CUDA gives you 163,840 threads, but you have to choreograph every one of them explicitly. Each model has its place.
02
Memory is the real bottleneck. Once you have enough threads, the limiting factor is almost always how fast you can feed them data. Transposing B, using shared memory, coalescing accesses : all memory tricks, not compute tricks.
03
The hardware matters. Block shape, warp alignment, shared memory size, register pressure, you can't optimize a GPU kernel without knowing how the hardware actually executes it. SIMT forced me to think at the hardware level.
04
Sometimes more parallelism hurts. V6 added loop unrolling and slightly regressed at scale, more registers meant fewer active warps, less latency hiding. The compiler was already doing a good job. Knowing when to stop is also a skill.
